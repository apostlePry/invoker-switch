# 并发控制基础模型设计 — 设计方案、好处与架构分析报告

> 项目：bhrpc-learning | 对应章节：第88章（并发控制基础模型设计）

---

## 一、功能概述与设计背景

### 1.1 问题背景

在 RPC 框架中，服务提供者需要同时处理大量并发请求。如果不做并发控制，所有请求的**业务逻辑处理**都在 Netty 的 I/O 线程（EventLoop）上执行，会导致：

- **I/O 线程阻塞**：业务逻辑中的耗时操作（数据库查询、远程调用等）会阻塞 Netty 的 I/O 线程，导致无法及时读取新请求和写入响应
- **吞吐量下降**：I/O 线程被业务逻辑占用后，新的请求无法被及时接收和处理
- **线程饥饿**：Netty 的 Worker EventLoopGroup 线程数有限（默认 CPU 核心数 × 2），全部被阻塞后整个服务不可用
- **无法弹性伸缩**：固定的 I/O 线程数无法根据负载动态调整并发处理能力

### 1.2 解决思路

将**网络 I/O 处理**与**业务逻辑计算**分离到不同的线程池：

- **Netty EventLoop 线程**：只负责网络读写（接收请求、发送响应），不执行任何业务逻辑
- **业务线程池**：负责执行反射调用、结果缓存等业务逻辑，线程数可配置、可弹性伸缩

这就是经典的 **Reactor 线程模型**中"主从 Reactor + 业务线程池"的架构。

## 二、项目中线程池体系总览

bhrpc 项目中存在 **三套线程池**，分别服务于不同场景：

| 线程池 | 模块 | 作用 | 创建方式 | 配置化 |
|--------|------|------|---------|--------|
| `ConcurrentThreadPool` | bhrpc-threadpool | **服务提供者业务线程池**（第88章新增） | 单例 + 参数化 | ✅ corePoolSize/maximumPoolSize 可配置 |
| `ClientThreadPool` | bhrpc-common | **服务消费者回调线程池** | 静态初始化 + 硬编码 | ❌ 固定 16/16/65536 |
| `ServerThreadPool` | bhrpc-common | **服务提供者线程池（旧版）** | 静态初始化 + 硬编码 | ❌ 固定 16/16/65536 |

此外还有两个**定时任务线程池**用于心跳：

| 线程池 | 位置 | 作用 |
|--------|------|------|
| `ScheduledExecutorService(2)` | `RpcConsumer` | 消费者端心跳：扫描不活跃连接 + 广播 ping |
| `ScheduledExecutorService(2)` | `BaseServer` | 提供者端心跳：扫描不活跃连接 + 广播 ping |

## 三、核心设计 — `ConcurrentThreadPool`

### 3.1 类定义

```java
// bhrpc-threadpool 模块
package io.binghe.rpc.threadpool;

public class ConcurrentThreadPool {
    private ThreadPoolExecutor threadPoolExecutor;
    private static volatile ConcurrentThreadPool instance;

    private ConcurrentThreadPool(){}

    private ConcurrentThreadPool(int corePoolSize, int maximumPoolSize){
        // 参数校验与修正
        if (corePoolSize <= 0) {
            corePoolSize = RpcConstants.DEFAULT_CORE_POOL_SIZE;      // 16
        }
        if (maximumPoolSize <= 0) {
            maximumPoolSize = RpcConstants.DEFAULT_MAXI_NUM_POOL_SIZE; // 16
        }
        if (corePoolSize > maximumPoolSize) {
            maximumPoolSize = corePoolSize;
        }
        this.threadPoolExecutor = new ThreadPoolExecutor(
            corePoolSize,                                          // 核心线程数
            maximumPoolSize,                                       // 最大线程数
            RpcConstants.DEFAULT_KEEP_ALIVE_TIME,                  // 600秒空闲保持
            TimeUnit.SECONDS,
            new ArrayBlockingQueue<Runnable>(RpcConstants.DEFAULT_QUEUE_CAPACITY)  // 65536容量
        );
    }
}
```

### 3.2 设计要点详解

#### 3.2.1 单例模式（双重检查锁 + volatile）

```java
public static ConcurrentThreadPool getInstance(int corePoolSize, int maximumPoolSize){
    if (instance == null){
        synchronized (ConcurrentThreadPool.class){
            if (instance == null){
                instance = new ConcurrentThreadPool(corePoolSize, maximumPoolSize);
            }
        }
    }
    return instance;
}
```

- **volatile**：`instance` 使用 `volatile` 修饰，防止指令重排序导致的"部分构造对象"问题
- **双重检查锁**：避免每次 `getInstance()` 都进入同步块，提升性能
- **参数只以第一次为准**：单例模式下，后续调用传入的 `corePoolSize`/`maximumPoolSize` 参数不会生效

#### 3.2.2 参数校验与自动修正

```java
if (corePoolSize <= 0)       → corePoolSize = 16     // 防止非法值
if (maximumPoolSize <= 0)    → maximumPoolSize = 16   // 防止非法值
if (corePoolSize > maximumPoolSize) → maximumPoolSize = corePoolSize  // 保证 core ≤ max
```

三重校验确保线程池参数始终合法，不会因配置错误导致 `ThreadPoolExecutor` 构造失败。

#### 3.2.3 ThreadPoolExecutor 核心参数

| 参数 | 值 | 含义 |
|------|-----|------|
| `corePoolSize` | 可配置，默认 16 | 核心线程数，始终存活的线程 |
| `maximumPoolSize` | 可配置，默认 16 | 最大线程数，负载高时临时扩容到此 |
| `keepAliveTime` | 600 秒 | 非核心线程空闲 600 秒后回收 |
| `workQueue` | `ArrayBlockingQueue(65536)` | 有界阻塞队列，容量 65536 |

**线程池行为**：
1. 请求到来时，若当前线程数 < corePoolSize，创建新线程处理
2. 当前线程数 = corePoolSize 时，请求放入队列
3. 队列满后，若当前线程数 < maximumPoolSize，创建非核心线程处理
4. 队列满且线程数 = maximumPoolSize，执行拒绝策略（默认 AbortPolicy）

#### 3.2.4 公开方法

```java
public void submit(Runnable task){
    threadPoolExecutor.submit(task);
}

public void stop(){
    threadPoolExecutor.shutdown();
}
```

- `submit()`：提交任务到线程池，内部调用 `ThreadPoolExecutor.submit()`
- `stop()`：优雅关闭线程池，内部调用 `shutdown()`（等待已提交任务完成）

## 四、使用方式 — 服务提供者端

### 4.1 `RpcProviderHandler` 中的使用

这是并发控制的核心应用场景——将业务逻辑从 Netty I/O 线程卸载到业务线程池：

```java
public class RpcProviderHandler extends SimpleChannelInboundHandler<RpcProtocol<RpcRequest>> {
    private final ConcurrentThreadPool concurrentThreadPool;

    public RpcProviderHandler(String reflectType, boolean enableResultCache,
                              int resultCacheExpire, int corePoolSize, int maximumPoolSize,
                              Map<String, Object> handlerMap){
        // ...
        this.concurrentThreadPool = ConcurrentThreadPool.getInstance(corePoolSize, maximumPoolSize);
    }

    @Override
    protected void channelRead0(ChannelHandlerContext ctx, RpcProtocol<RpcRequest> protocol)
            throws Exception {
        // ★ 关键：将业务逻辑提交到线程池，不在 Netty I/O 线程上执行
        concurrentThreadPool.submit(() -> {
            RpcProtocol<RpcResponse> responseRpcProtocol = handlerMessage(protocol, ctx.channel());
            ctx.writeAndFlush(responseRpcProtocol).addListener(new ChannelFutureListener() {
                @Override
                public void operationComplete(ChannelFuture channelFuture) throws Exception {
                    logger.debug("Send response for request " + protocol.getHeader().getRequestId());
                }
            });
        });
    }
}
```

**核心流程**：

```
Netty I/O 线程                          业务线程池
    │                                      │
    │ channelRead0(protocol)               │
    │─────────────────────────────────────>│
    │  concurrentThreadPool.submit(task)   │
    │                                      │ handlerMessage(protocol, channel)
    │                                      │   ├─ 心跳消息处理
    │                                      │   └─ 请求消息处理
    │                                      │       ├─ 结果缓存检查
    │                                      │       ├─ 反射调用真实方法
    │                                      │       └─ 构造响应
    │                                      │
    │  ctx.writeAndFlush(response)  <──────│
    │                                      │
    │  响应写回客户端                        │
```

### 4.2 参数传递链路

线程池参数从配置层到 `ConcurrentThreadPool` 的完整传递链路：

```
yml/properties 配置
    │
    ▼
SpringBootProviderConfig (corePoolSize, maximumPoolSize)
    │
    ▼
SpringBootProviderAutoConfiguration → new RpcSpringServer(..., corePoolSize, maximumPoolSize)
    │
    ▼
BaseServer (corePoolSize, maximumPoolSize)
    │
    ▼
BaseServer.startNettyServer() → new RpcProviderHandler(..., corePoolSize, maximumPoolSize)
    │
    ▼
RpcProviderHandler → ConcurrentThreadPool.getInstance(corePoolSize, maximumPoolSize)
    │
    ▼
new ThreadPoolExecutor(corePoolSize, maximumPoolSize, 600, SECONDS, new ArrayBlockingQueue<>(65536))
```

## 五、消费者端线程池 — `ClientThreadPool`

### 5.1 定义

```java
// bhrpc-common 模块
public class ClientThreadPool {
    private static ThreadPoolExecutor threadPoolExecutor;

    static {
        threadPoolExecutor = new ThreadPoolExecutor(16, 16, 600L, TimeUnit.SECONDS,
            new ArrayBlockingQueue<Runnable>(65536));
    }

    public static void submit(Runnable task){
        threadPoolExecutor.submit(task);
    }

    public static void shutdown() {
        threadPoolExecutor.shutdown();
    }
}
```

### 5.2 使用场景

`ClientThreadPool` 在消费者端用于**异步回调执行**：

```java
// RPCFuture.runCallback()
private void runCallback(final AsyncRPCCallback callback) {
    final RpcResponse res = this.responseRpcProtocol.getBody();
    ClientThreadPool.submit(() -> {
        if (!res.isError()) {
            callback.onSuccess(res.getResult());
        } else {
            callback.onException(new RuntimeException("Response error", new Throwable(res.getError())));
        }
    });
}
```

**设计意图**：异步回调不应在 Netty I/O 线程上执行，避免用户回调中的耗时逻辑阻塞网络读写。

### 5.3 关闭时机

```java
// RpcConsumer.close()
public void close(){
    RpcConsumerHandlerHelper.closeRpcClientHandler();
    eventLoopGroup.shutdownGracefully();
    ClientThreadPool.shutdown();    // ← 关闭消费者线程池
    executorService.shutdown();
}
```

### 7.3 消费者 vs 提供者的线程模型差异

| 维度 | 服务提供者 | 服务消费者 |
|------|-----------|-----------|
| Netty EventLoop 数 | 默认 2N（CPU核心×2） | 固定 4 |
| I/O 线程是否执行业务 | ❌ 不执行，提交到业务线程池 | ✅ 直接在 I/O 线程完成响应匹配 |
| 业务线程池 | `ConcurrentThreadPool`（可配置） | `ClientThreadPool`（硬编码） |
| 业务线程池用途 | 执行反射调用 + 结果缓存 | 执行异步回调 |
| 心跳线程池 | `ScheduledExecutorService(2)` | `ScheduledExecutorService(2)` |

**设计差异的原因**：

- **提供者端**：业务逻辑（反射调用）可能很耗时，必须从 I/O 线程卸载
- **消费者端**：响应匹配（`pendingRPC.remove → rpcFuture.done`）非常轻量，可以在 I/O 线程直接完成；只有用户注册的异步回调可能耗时，需要卸载到 `ClientThreadPool`

## 十、设计好处分析

### 10.1 I/O 与业务逻辑解耦 — 核心收益

| 维度 | 无业务线程池 | 有业务线程池 |
|------|------------|------------|
| I/O 线程职责 | 网络读写 + 业务逻辑 | 仅网络读写 |
| 业务耗时影响 | 阻塞 I/O 线程，影响所有连接 | 仅占用业务线程，I/O 线程不受影响 |
| 吞吐量 | 受限于 I/O 线程数 × 业务处理速度 | I/O 线程持续接收请求，业务线程并行处理 |
| 尾部延迟 | 一个慢请求阻塞 I/O 线程，后续请求排队 | 慢请求仅占用一个业务线程 |

**量化示例**：假设 4 个 I/O 线程，业务逻辑平均耗时 10ms
- **无业务线程池**：最大吞吐 ≈ 4 / 0.01 = 400 QPS
- **有业务线程池（16线程）**：最大吞吐 ≈ 16 / 0.01 = 1600 QPS

### 10.2 可配置的并发度

`corePoolSize` 和 `maximumPoolSize` 可通过配置文件调整，无需修改代码：

- **CPU 密集型业务**：设置 corePoolSize = CPU 核心数 + 1
- **I/O 密集型业务**：设置 corePoolSize = CPU 核心数 × 2 或更高
- **弹性伸缩**：corePoolSize < maximumPoolSize 时，线程池在负载高峰自动扩容，低峰自动回收

### 10.3 背压控制

`ArrayBlockingQueue(65536)` 提供了天然的背压机制：

- 队列未满时：请求排队等待核心线程处理
- 队列满后：创建非核心线程（若未达 maximumPoolSize）
- 队列满且线程数达上限：触发拒绝策略，防止 OOM

### 10.4 单例复用

`ConcurrentThreadPool` 采用单例模式，整个服务提供者 JVM 中只有一个业务线程池实例，避免：
- 多个 Handler 各自创建线程池导致的资源浪费
- 线程数不可控的问题
- 线程池间负载不均衡的问题

### 10.5 参数自动修正

三重校验确保线程池参数始终合法：
- `corePoolSize <= 0` → 使用默认值 16
- `maximumPoolSize <= 0` → 使用默认值 16
- `corePoolSize > maximumPoolSize` → `maximumPoolSize = corePoolSize`

即使配置出错，框架也不会崩溃，而是降级到安全默认值。

### 10.6 消费者端回调线程池隔离

`ClientThreadPool` 将用户回调从 Netty I/O 线程卸载：
- 用户回调中可能包含耗时逻辑（日志记录、结果转换、二次调用等）
- 如果在 I/O 线程执行，会阻塞响应接收
- 隔离后，I/O 线程只做轻量的 `pendingRPC.remove → rpcFuture.done`，回调在独立线程池执行

| 层次 | 类名 | 职责 |
|------|------|------|
| **线程池模块** | `ConcurrentThreadPool` | 可配置的业务线程池，单例模式，用于提供者端业务逻辑处理 |
| **通用模块** | `ClientThreadPool` | 消费者端回调线程池，硬编码参数，用于异步回调执行 |
| **通用模块** | `ServerThreadPool` | 旧版提供者端线程池，硬编码参数（已被 ConcurrentThreadPool 替代） |
| **常量** | `RpcConstants` | 线程池默认参数：corePoolSize=16, maximumPoolSize=16, keepAlive=600s, queueCapacity=65536 |
| **提供者Handler** | `RpcProviderHandler` | 在 channelRead0 中将业务逻辑提交到 ConcurrentThreadPool |
| **提供者Server** | `BaseServer` | 持有 corePoolSize/maximumPoolSize，传递给 RpcProviderHandler |
| **消费者Future** | `RPCFuture` | 在 runCallback 中将回调提交到 ClientThreadPool |
| **消费者** | `RpcConsumer` | 关闭时调用 ClientThreadPool.shutdown() |
| **Spring Boot** | `SpringBootProviderConfig` | 持有 corePoolSize/maximumPoolSize yml配置 |
| **Spring Boot** | `SpringBootProviderAutoConfiguration` | 将配置传递给 RpcSpringServer |

## 十二、潜在优化点与改进建议

### 12.1 自定义 ThreadFactory

当前使用 `ThreadPoolExecutor` 默认的 ThreadFactory，线程名为 `pool-N-thread-M`，不利于问题排查。建议：

```java
ThreadFactory namedFactory = new ThreadFactoryBuilder()
    .setNameFormat("bhrpc-business-%d")
    .setDaemon(true)
    .build();
```

### 12.2 自定义 RejectedExecutionHandler

当前使用默认的 `AbortPolicy`，队列满时直接抛出 `RejectedExecutionException`。建议：

- **CallerRunsPolicy**：队列满时由调用线程（Netty I/O 线程）执行，提供背压
- **自定义策略**：记录被拒绝的请求信息，触发告警，或降级返回默认响应

### 12.3 线程池监控

建议增加线程池运行指标暴露：

- 活跃线程数、队列大小、已完成任务数、拒绝任务数
- 通过 JMX 或 Micrometer 暴露给监控系统

### 12.4 消费者端线程池可配置化

`ClientThreadPool` 当前参数硬编码（16/16/65536），建议也支持配置化，与 `ConcurrentThreadPool` 保持一致。

### 12.5 动态线程池

当前线程池参数在启动时确定，运行期不可调整。建议：

- 接入配置中心（Nacos Config），支持运行时动态调整 corePoolSize/maximumPoolSize
- `ThreadPoolExecutor` 原生支持 `setCorePoolSize()` 和 `setMaximumPoolSize()` 方法

### 12.6 多线程池隔离

当前所有服务共享一个 `ConcurrentThreadPool`。建议：

- 按服务分组（group）或接口创建独立线程池
- 核心服务分配更多线程，非核心服务限制线程数
- 避免一个慢服务占满线程池影响其他服务

### 12.7 队列容量可配置

`ArrayBlockingQueue` 容量硬编码为 65536，建议也支持配置化，不同业务场景对队列深度需求不同。

## 十四、总结

bhrpc 的并发控制基础模型通过 **I/O 线程与业务线程池分离** 的架构，实现了网络读写与业务逻辑的解耦，核心设计亮点包括：

1. **Reactor + 业务线程池架构**：Netty EventLoop 只做网络 I/O，业务逻辑在 `ConcurrentThreadPool` 中执行，互不阻塞
2. **可配置的并发度**：`corePoolSize` 和 `maximumPoolSize` 通过 yml/properties/XML/注解/原生 API 五种方式配置
3. **单例复用**：全局一个业务线程池实例，避免资源浪费和负载不均
4. **参数自动修正**：三重校验确保线程池参数始终合法，配置出错也不崩溃
5. **消费者端回调隔离**：`ClientThreadPool` 将用户回调从 I/O 线程卸载，避免回调逻辑阻塞网络读写
6. **背压控制**：`ArrayBlockingQueue(65536)` 提供有界队列，防止请求堆积导致 OOM
7. **新旧兼容**：`ConcurrentThreadPool` 是 `ServerThreadPool` 的增强替代，参数可配置，旧版保留兼容

该并发模型是高性能 RPC 框架的基础设施，确保了在高并发场景下服务提供者的稳定性和吞吐量。
